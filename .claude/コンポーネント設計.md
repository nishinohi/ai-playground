# コンポーネント設計

基本は Atomic Design パターンを採用するが以下に解説する独自の要素を組み込んだコンポーネント設計であることに注意。
以下のディレクトリ構成とする。

```txt
app
├── routes
│   └── [file system routing by react router]
└── components
    ├── atoms
    ├── molecules
    ├── organisms
    ├── templates
    └── domains
```

`routes` 配下には React Router の file system routing に従ったコンポーネントを作成する。
`atoms`, `molecules`, `organisms`, `templates` は通常の Atomic Design のルールに準拠する。
React Router では `routes` ディレクトリにあるファイルが各画面に対応するため `pages` は省略する。
`routes`にあるファイル基本的に`templates` のコンポーネントに loader から取得したデータを渡すだけのシンプルな実装となる。

## routes の構成と templates の読み込む

file system routing に従って `routes` に配置するコンポーネントに実装本体は配置しない。
`routes` に配置するコンポーネントは loader、action のサーバーサイドの実装と、meta などの必要なメタ情報を実装する。
コンポーネントの実装は以下の命名規則に従った `templates` 配下のコンポーネントを読み込む。

/customer-list というパスの画面を実装する例。

```typescript
// app/routes/customer-list.tsx
export async function action({ request }: Route.ActionArgs) {
     // 省略
}
export async function loader({ request }: Route.LoaderArgs) {
     // 省略
}

// コンポーネント名はパスのキャメルケースとする
export default function CustomerList() {
     return (<CustomerListTemplate />)
}

// app/components/templates/CustomerList/index.tsx
// ディレクトリ名はパスのキャメルケースとする
// コンポーネント名はパスのキャメルケースの末尾に Template をつけるx
export const CustomerListTemplate() {
     return // コンポーネントの実装
}
```

## templates の構成

それぞれの画面の実装は `templates` に以下のディレクトリ構成で配置する。
また、画面に依存するロジックを持つコンポーネントや、特定の画面のみで利用するコンポーネントも該当する画面の `templates` に配置する。
全てのコンポーネントは自身の名称のディレクトリの中にコンポーネントの実態、スタイリング、Unit などコンポーネントに関する全ての実装を含む。

```txt
templates
└── [TemplateName]               ## routes のファイル名と対応する名称
     ├── index.tsx               ## 画面のメイン実装
     ├── [ComponentName]         ## templates に依存したコンポーネント
     │   ├── index.tsx           ## コンポーネントの実装
     │   ├── index.test.tsx      ## コンポーネントの Unit テスト
     │   └── styles.module.css   ## コンポーネントのスタイリング（省略可）
     ├── [ComponentName]
     │   ├── index.tsx
     │   ├── index.test.tsx
     │   └── styles.module.css
     ├── hooks                   ## 画面に依存した hook の格納場所
     │   ├── [hookName].ts       ## hook の実装
     │   └── [hookName].ts
     └── index.test.tsx          ## template の Unit テスト
```

## atoms, molecules, organisms の運用

`atoms`, `molecules`, `organisms` に配置するコンポーネントは以下のルールを厳守する。

- 画面に依存するロジックを一切含まないように実装する
- 画面依存のロジックを含んでいない場合でも特定の画面だけで使用する前提のコンポーネントは templates 内の個別の画面に配置する
- shadcn/ui から取り込んだコンポーネントは粒度に応じて`atoms`, `molecules`, `organisms`のいずれかに配置する

## atoms, molecules, organisms のディレクトリ構成

各ディレクトリのコンポーネントのディレクトリ構成は以下とする。

```txt
[atoms, molecules, organisms]
 └── [ComponentName]
      ├── index.tsx           ## コンポーネントの実装
      ├── index.test.tsx      ## コンポーネントの Unit テスト
      └── styles.module.css   ## コンポーネントのスタイリング（省略可）
```

## domains の運用

画面依存を含むコンポーネントだが、複数の画面で同じコンポーネントを再利用したい場合は `domains` ディレクトリを利用する。
コンポーネント名は画面依存が存在していることがわかるような命名を基本とする。構成は以下の通り。

```txt
domains
└── [ComponentName]
     ├── index.tsx           ## コンポーネントの実装
     ├── index.test.tsx      ## コンポーネントの Unit テスト
     └── styles.module.css   ## コンポーネントのスタイリング（省略可）
```

## コンポーネントのプロパティの型名称

コンポーネントの実装においてメインコンポーネントが受け取るプロパティの型には`type`を使用し型の名称は`Props`で統一する。
メインコンポーネントから使用されるサブコンポーネントを同一ファイル内で実装している場合、サブコンポーネントの型名称はこのルールには従わなくて良い。
