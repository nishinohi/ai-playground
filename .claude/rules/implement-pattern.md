# 実装パターン

## コードスタイル

- 早期リターンを重視
- 小さく始めて段階的に拡張
- 過度な抽象化を避ける
- コードよりも型を重視
- 複雑さに応じてアプローチを調整
- 関数優先（クラスは必要な場合のみ）
- 不変更新パターンの活用
- エラーとユースケースの列挙型定義
- 1ファイル500行以内を目標にする

## 型定義

型定義には原則として`type`を使用する。`interface`は拡張性が必要となる型やサードパーティライブラリで必要とされる場合以外では使用しない。

Branded 型で型安全性を確保する。

```typescript
type Branded<T, B> = T & { _brand: B }
type Money = Branded<number, 'Money'>
type Email = Branded<string, 'Email'>
```

## 値オブジェクト

- 不変
- 値に基づく同一性
- 自己検証
- ドメイン操作を持つ

```typescript
// 作成関数はバリデーション付き
function createMoney(amount: number): Result<Money, Error> {
  if (amount < 0) return err(new Error('負の金額不可'))
  return ok(amount as Money)
}
```

## エンティティ

- IDに基づく同一性
- 制御された更新
- 整合性ルールを持つ

## リポジトリ

- ドメインモデルのみを扱う
- 永続化の詳細を隠蔽
- テスト用のインメモリ実装を提供

## アダプターパターン

- 外部依存を抽象化
- インターフェースは呼び出し側で定義
- テスト時は容易に差し替え可能

## 実装の手順

### 型設計

- まず型を定義
- ドメインの言語を型で表現

### 純粋関数から実装

- 外部依存のない関数を先に
- テストを先に書く

### 副作用を分離

- IO操作は関数の境界に押し出す
- 副作用を持つ処理をPromiseでラップ

### アダプター実装

- 外部サービスやDBへのアクセスを抽象化
- テスト用モックを用意
